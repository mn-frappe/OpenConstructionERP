# Часть 0. Как пользоваться этим документом

## 0.1 Жёсткие правила работы Claude Code

1. **Никогда не пиши код до завершения Части 1 (Research).** Это самая частая ошибка. Сначала исследование, потом проектирование, потом код.
2. **Каждое утверждение о структуре кода должно быть подтверждено `view` или `grep` команды.** Не предполагай, что код устроен так-то — проверяй.
3. **Если ты не нашёл существующий паттерн — спроси, не выдумывай новый.** Из 10 случаев, когда хочется ввести «свой» подход, в 9 уже есть устоявшийся в проекте.
4. **Все изменения должны быть минимальными.** Меняй только то, что нужно для задачи. Не «причёсывай» проходя мимо.
5. **Если задача застряла дольше 30 минут на одном месте — остановись и переформулируй.** Скорее всего ты идёшь не в ту сторону.

## 0.2 Что в этом документе универсально, а что специфично

- **Части 0–3, 5–7** — универсальные правила. Применяются ко всем 18 модулям без исключения.
- **Часть 4** — индивидуальные спецификации каждого модуля. Раздел берётся в работу пакетом по одному модулю.

## 0.3 Глоссарий

- **OCE** — OpenConstructionERP, целевая платформа.
- **Module** — функциональная единица (Service, HSE, QMS, …).
- **Entity** — сущность данных (Project, Contract, Subcontractor, …).
- **PRD** — Product Requirements Document (раздел в Части 4 для каждого модуля).
- **QA Wave** — одна волна проверки качества (см. Часть 3).
- **Definition of Done (DoD)** — формальный список того, что должно быть сделано, чтобы задача считалась завершённой.

# Часть 1. Обязательная фаза предварительного исследования

> **Эта фаза занимает 20–40% времени работы над модулем. Не сокращай её.** Сэкономленное здесь время в 5 раз дороже стоит исправлением неверной архитектуры.

## 1.1 Исследование кодовой базы (Codebase Discovery)

Claude Code обязан прежде всего прочитать и зафиксировать:

### 1.1.1 Стек технологий и инструменты
- `package.json` (frontend и backend, если разделены) — все версии зависимостей.
- `pyproject.toml` / `requirements.txt` / `poetry.lock` — для Python-части.
- `tsconfig.json`, `eslintrc`, `prettierrc` — конфигурация языка и стиля.
- Файлы сборки и CI: `Dockerfile`, `docker-compose.yml`, `.github/workflows/*`, `Makefile`.
- Скрипты в `package.json` / `Makefile` для запуска, тестов, миграций.

### 1.1.2 Структура репозитория
- Запусти `view` на корне репозитория.
- Идентифицируй монорепо vs одиночный проект.
- Найди границы пакетов (apps/, packages/, services/, libs/).
- Найди `README.md` и любые `CLAUDE.md`, `.cursorrules`, `AGENTS.md`, `CONTRIBUTING.md` — прочти их полностью.

### 1.1.3 Архитектурные слои
- **Backend**: где роутеры, где сервисы, где модели данных, где миграции, где фоновые задачи.
- **Frontend**: где роутер, где страницы, где shared-компоненты, где state-менеджер, где API-клиент.
- **БД**: ORM или сырой SQL, какие схемы, какие миграции.

### 1.1.4 Аутентификация и авторизация
- Где зарегистрированы пользователи.
- Как устроены роли и права (RBAC).
- Как проверяется доступ к ресурсам.
- Есть ли row-level security.

### 1.1.5 Общая инфраструктура
- Как устроена работа с файлами (S3/локальная, как генерируются presigned URL).
- Как устроена очередь фоновых задач (Celery, RQ, Bull, …).
- Как отправляются email/push/SMS-уведомления.
- Как устроен поиск (Postgres FTS, Elasticsearch, Qdrant, …).
- Как устроен i18n.
- Как устроено логирование, трассировка, метрики.

## 1.2 Инвентаризация существующих модулей OCE

Claude Code должен сформировать **краткую карту существующих модулей** — это нужно, чтобы понять, куда подключаться. Для каждого модуля:

| Что фиксировать | Зачем |
|---|---|
| Точка входа (роут URL, файл страницы) | Для добавления своих ссылок и навигации |
| Главная сущность (модель данных) | Для построения связей |
| API endpoints | Для интеграции |
| Permissions / scopes | Чтобы корректно отрабатывать RBAC |
| Главные UI-компоненты | Чтобы переиспользовать, а не дублировать |

Сохрани результат в `docs/integration/existing-modules-map.md` — он пригодится во всех будущих модулях.

## 1.3 Документирование паттернов проекта

Прежде чем писать что-то новое, **обязательно** найди в коде образец того же:

- Форма создания/редактирования сущности — какая структура, какая валидация, как обрабатываются ошибки.
- Список (таблица) с фильтрами/поиском/сортировкой/пагинацией — какой компонент, как параметризуется.
- Страница деталей сущности — какой layout, как располагаются вкладки/timeline/боковая панель.
- Модальные окна и confirmation dialogs.
- Toast/notification UI.
- Loading states (skeleton screens, спиннеры).
- Empty states (когда данных нет).
- Error states (когда что-то сломалось).
- Загрузка файлов.
- Просмотр галереи изображений.
- Карты (если есть).
- Календари.
- Drag-and-drop.
- Audit log в UI.
- Activity timeline.

Этот «реестр паттернов» зафиксируй в `docs/integration/patterns.md`. В реализации модуля используй **только** найденные паттерны. Если на нужный сценарий паттерна нет — спроси у автора, а не выдумывай.

## 1.4 Реальные данные для разработки

- Узнай, есть ли seed/демо-данные в репозитории. Если есть — запусти и убедись, что приложение работает на них.
- Если нет — реализация модуля **должна включать** seed-данные (см. Часть 2.4).

## 1.5 Артефакты фазы исследования

К концу Части 1 у Claude Code должны существовать:

1. `docs/integration/existing-modules-map.md` — карта существующих модулей.
2. `docs/integration/patterns.md` — реестр UI и кодовых паттернов проекта.
3. **План реализации модуля** (см. шаблон в Части 7) — записан в `TASK.md` или комментарии PR.
4. **Список вопросов**, на которые без ответа нельзя начинать (если есть) — записан в `QUESTIONS.md`. Ответы получены до старта кодирования.

# Часть 2. Универсальные стандарты качества реализации

## 2.1 Жёсткие запреты (НЕ нарушаются никогда)

| ❌ Запрещено | Почему |
|---|---|
| Хардкод значений (URL, цены, лимиты, ставки) | Должно быть в конфиге, env-переменных или БД |
| Mock-данные в production-коде | Mock допустим только в тестах |
| Заглушки `// TODO`, `pass`, `throw new Error("Not implemented")` без зарегистрированного follow-up issue | Незаконченный код в проде |
| Тип `any` / `unknown` без явного обоснования в комментарии | Теряется безопасность типов |
| Закомментированный код | Удалить или вернуть, но не оставлять «на всякий случай» |
| Дублирование логики между фронтом и бэком, или между модулями | Извлекается в общий пакет/утилиту |
| Новый UI-паттерн, когда существующий работает | Каждый новый паттерн — техдолг |
| Новая система состояния, когда есть существующая | Хаос в состоянии — главная причина багов |
| Inline-стили (`style={{...}}`), когда есть CSS-классы | Несовместимо с темизацией |
| Прямой `fetch` / `axios`, минуя единый API-клиент проекта | Теряется обработка ошибок, токенов, ретраев |
| Прямой `console.log` в production-коде | Используй центральный логгер |
| Изменение публичного API существующего модуля без миграционного плана | Может сломать клиентов |
| Сырые SQL-запросы вместо ORM, когда есть ORM | Несовместимо с миграциями и тестами |
| Использование данных пользователя в запросах без валидации | SQL/NoSQL-injection, XSS |
| Хранение секретов в коде или в репозитории | Утечки |

## 2.2 Обязательные элементы каждой функциональности

Для **каждой** реализованной user-story (даже простой) должно быть:

- ✅ **RBAC-интеграция** — проверка прав на каждое действие (фронт **и** бэк; UI скрывает кнопки без права, API возвращает 403).
- ✅ **Audit log** — каждое значимое действие записывается в журнал (кто, что, когда, что было, что стало).
- ✅ **i18n** — каждая строка для пользователя идёт через i18n-ключ, не хардкодится.
- ✅ **Состояния UI** — loading state (skeleton), empty state (с CTA «создать первое»), error state (читаемый текст).
- ✅ **Confirmation dialogs** — для удаления и других необратимых действий.
- ✅ **Валидация** — клиентская **и** серверная (никогда только клиентская).
- ✅ **Пагинация** — все списки длиннее 50 элементов (server-side).
- ✅ **Поиск и фильтры** — все списки.
- ✅ **Сортировка колонок** — все таблицы.
- ✅ **Экспорт** — Excel/CSV для всех значимых списков.
- ✅ **Mobile responsive** — все экраны (минимум 360px width).
- ✅ **Keyboard accessible** — Tab/Esc/Enter работают корректно.
- ✅ **A11y** — корректные ARIA-роли, alt-тексты на изображениях, контраст.
- ✅ **Транзакции БД** — для операций, изменяющих несколько таблиц.
- ✅ **Фоновые задачи** — для тяжёлых операций (OCR, отчёты, импорты).
- ✅ **Уведомления** — email/in-app для событий, требующих внимания пользователя.
- ✅ **Миграции БД** — каждое изменение схемы — отдельная миграция с up и down.
- ✅ **Seed-данные** — реалистичные демо-данные для разработки и QA.
- ✅ **OpenAPI / GraphQL schema** — для всех API — автодокументация.
- ✅ **TypeScript-типы** — генерируются из API-схемы (не дублируются вручную).
- ✅ **Тесты** — unit, integration, e2e (см. Часть 3).
- ✅ **Documentation** — каждая новая фича добавлена в `docs/`.

## 2.3 Что обязательно при работе с существующими модулями

- ✅ **Сохранение обратной совместимости.** Старые роуты, API, БД-структуры работают как раньше.
- ✅ **Миграция данных, если требуется.** Если расширяешь модель — пишешь миграцию, которая корректно поднимает старые записи в новую схему.
- ✅ **Тестирование старых сценариев.** Перед слиянием убедись, что существующая функциональность модуля не сломалась.
- ✅ **Сохранение URL.** Если переименовываешь раздел, оставляешь редиректы со старого URL.

## 2.4 Seed-данные и демо-режим

Каждый реализованный модуль обязан предоставлять:

- Скрипт `seed:<module>` (например, `seed:service`, `seed:hse`), который создаёт реалистичный набор данных:
  - Несколько проектов разного типа и статуса.
  - Несколько сотен записей в основной сущности модуля (не 3, не 5 — а 50–200 для проверки списков, фильтров, пагинации).
  - Связанные данные в смежных модулях.
  - Покрытие крайних случаев (записи в разных статусах, с истёкшими сроками, с прикреплёнными файлами).
- Все seed-данные **детерминированы** (одинаковый seed даёт одинаковые данные).
- Все seed-данные **локализованы** (имена, адреса, телефоны — корректные для нескольких языковых регионов).
- Возможность **очистить и пере-сидить** одной командой.

## 2.5 Конфигурация и переменные окружения

- Все настройки, которые могут отличаться между средами, идут в `.env`.
- Каждая новая переменная окружения добавляется в `.env.example` с комментарием.
- Чтение env-переменных — через единый конфиг-модуль, не разбросано по коду.

## 2.6 Логирование, метрики, трассировка

- Используется существующий логгер проекта; уровни (debug/info/warn/error) корректные.
- Структурированные логи (JSON), не plain text, в production.
- Бизнес-метрики (счётчики важных событий) экспонируются через тот же механизм, что и в существующих модулях.
- Корреляционный ID (trace_id) сохраняется во всех логах одного запроса.

# Часть 3. Протокол многоволновой проверки качества (QA)

> **Модуль не считается готовым, пока все 7 волн не пройдены и не задокументированы.** Это не формальность. Каждая волна ловит свой класс багов.

## Волна 1. Статический анализ (Static Analysis)

**Цель:** найти синтаксические, типовые и структурные ошибки до запуска.

**Команды:**
- `npm run lint` (или эквивалент проекта) — ESLint, Prettier.
- `npm run typecheck` (или `tsc --noEmit`) — TypeScript / Pyright / mypy.
- `npm run build` — сборка должна пройти без ошибок и warning'ов.
- `npm run analyze` (если есть) — bundle-size analysis для frontend.
- Любые дополнительные статические анализаторы проекта (Semgrep, Bandit, Snyk).

**Критерии прохождения:**
- 0 ошибок типов.
- 0 ESLint ошибок (warning'и допустимы только в legacy-коде, не в новом).
- Размер bundle вырос не более чем на 5% (если больше — обосновать).
- Нет нового мёртвого кода.

## Волна 2. Unit-тесты

**Цель:** проверить чистые функции, утилиты, хуки изолированно.

**Покрытие:** все новые pure-функции, бизнес-логика, утилиты, кастомные хуки.

**Критерии прохождения:**
- 100% тестов проходит.
- Coverage новых модулей ≥ 80% (если в проекте установлен другой порог — берём его).
- Тесты покрывают edge cases: null, undefined, пустой массив, очень большое число, специальные символы, разные локали.

## Волна 3. Integration-тесты

**Цель:** проверить интеграцию слоёв (API + БД, фронт + API, фронт + state).

**Покрытие:**
- Каждый API endpoint модуля.
- Каждое значимое БД-взаимодействие.
- Проверка RBAC: запрос без права возвращает 403, с правом — 200.
- Транзакционность (если что-то сломалось посередине — откат).

**Критерии прохождения:**
- Все integration-тесты проходят на чистой БД.
- Все integration-тесты проходят на БД с seed-данными.

## Волна 4. End-to-End браузерные тесты со скриншотами

**Цель:** проверить реальные пользовательские сценарии в браузере, как видит их пользователь.

**Инструменты:** Playwright (предпочтительно) или Cypress, если есть в проекте.

**Что делать:**
1. Для каждого сценария из раздела `🧪 Test scenarios for browser QA` в PRD модуля — написать e2e-тест.
2. На ключевых шагах сценария — `await page.screenshot({ path: ... })`.
3. Скриншоты сохраняются в `docs/qa/screenshots/<module>/<scenario>/<step>.png`.
4. Тесты запускаются на трёх viewport'ах: desktop (1440×900), tablet (768×1024), mobile (375×667).
5. Каждый сценарий проигрывается с реальным сидом БД, без mock'ов.

**Критерии прохождения:**
- Все сценарии модуля проходят на всех трёх viewport'ах.
- Скриншоты сохранены, читаемы и соответствуют ожидаемому состоянию.
- Базовые проверки: нет console errors во время теста, нет network errors (4xx/5xx), время загрузки страницы < 3 сек на dev-машине.

## Волна 5. Кросс-модульная интеграционная верификация

**Цель:** убедиться, что новый модуль корректно подключён к остальной платформе.

**Проверки:**
- Навигация: новые пункты меню есть, ведут куда нужно, скрыты у пользователей без прав.
- Глобальный поиск: ищет по новым сущностям.
- Главный Dashboard: отражает новые данные (если применимо).
- BI Dashboards: KPI нового модуля доступны.
- Audit log центральный: записи нового модуля видны и фильтруются.
- Notifications: события модуля попадают в центральную ленту уведомлений.
- Mobile App (если модуль имеет мобильную часть): синхронизация работает.
- Permissions matrix: роли корректно настроены, в админке прав видны новые scopes.
- Связи с конкретными модулями (указаны в разделе `🧩 Integration points` каждого PRD) — все работают.

## Волна 6. Ручной smoke-тест с фотофиксацией

**Цель:** глазами проверить «как оно живое».

**Что делать:**
1. Открыть приложение в браузере.
2. Войти под разными ролями (минимум: admin, project manager, foreman, accountant, читатель).
3. Пройти все главные сценарии из PRD модуля.
4. На каждом ключевом экране — скриншот в `docs/qa/manual-screenshots/<module>/<role>/<screen>.png`.
5. Сравнить визуально с теми, что сделал Playwright (Волна 4).

## 7.1 Шаблон TASK.md для одной сессии

При старте новой сессии разработки одного модуля:

```markdown

## Контекст
- Корневой документ: CLAUDE.md (мастер-инструкция)
- PRD модуля: см. Часть 4, раздел "Модуль N" в CLAUDE.md
- Существующие модули: см. docs/integration/existing-modules-map.md
- Паттерны проекта: см. docs/integration/patterns.md

## План (заполняет Claude Code до начала кодирования)

### 1. Research phase findings
[Записать ответы на research questions модуля]

### 2. Integration plan
[Зафиксировать конкретные точки интеграции с существующими модулями]

### 3. Architecture decisions
[Перечислить нетривиальные архитектурные решения с обоснованием]

### 4. Implementation phases
[Уточнённый список шагов из PRD, конкретизированный под нашу кодовую базу]

### 5. Definition of Done
[Чек-лист из PRD + специфика этого спринта]

### 6. Test plan
[Сценарии из PRD + новые если выявлены при анализе]

### 7. Risks & open questions
[Что может пойти не так, что неясно]

## Прогресс
- ☐ Phase 0: Research done
- ☐ Phase 1: Data model + migrations
- ☐ Phase 2: API layer
- ☐ Phase 3: UI desktop
- ☐ Phase 4: UI mobile (if applicable)
- ☐ Phase 5: Integration with other OCE modules
- ☐ Phase 6: Seed data
- ☐ Phase 7: i18n, audit, notifications
- ☐ Phase 8: Background jobs

## QA Waves
- ☐ Wave 1: Static analysis
- ☐ Wave 2: Unit tests
- ☐ Wave 3: Integration tests
- ☐ Wave 4: E2E browser tests with screenshots
- ☐ Wave 5: Cross-module integration verification
- ☐ Wave 6: Manual smoke test
- ☐ Wave 7: Final sign-off
```

## 7.2 Порядок шагов в сессии

1. **Старт.** Прочитать корневой `CLAUDE.md` целиком.
2. **Research.** Выполнить Часть 1 для текущего модуля.
3. **Plan.** Заполнить раздел «План» в `TASK.md` до начала кодирования.
4. **Review (опционально).** Если есть человек на review — показать план, получить ack.
5. **Code.** Реализовать по фазам, отмечая ☐ → ☑ в `TASK.md`.
6. **QA.** Прогонять волны параллельно реализации, не оставлять на конец.
7. **Sign-off.** Заполнить Wave 7 чек-лист.
8. **Merge.** PR / merge.
9. **Cleanup.** Обновить `existing-modules-map.md` с новым модулем.

## 7.3 Что делать, если что-то идёт не так

- **Архитектура не складывается** → пауза, переписать раздел «Architecture decisions» в TASK.md, обосновать новый подход.
- **Тест падает и непонятно почему** → не «обходить» тест, не подгонять реализацию под тест; разобраться в первопричине.
- **Появилась идея «причесать заодно»** → НЕТ. Создать issue, продолжить только основную задачу.
- **Кажется, что-то противоречит правилам в CLAUDE.md** → НЕТ, не нарушать; уточнить у автора, обновить CLAUDE.md формально.
- **Затронут существующий модуль** → провести регрессию этого модуля до закрытия задачи.

# Заключение и что обычно забывают на уровне всего проекта

Я ничего не «забыл» — описал максимально подробно. Но **что часто упускают** даже опытные команды при таком масштабе:

1. **Disaster recovery plan.** Что делаем, если упадёт продакшен. Регулярные drill'ы.
2. **Onboarding для новых пользователей.** Без отдельного flow «первый вход» — пользователь не справится с 22 модулями.
3. **Migration tooling из конкурентов.** Если хотим, чтобы клиент мигрировал — нужны импортёры.
4. **Performance benchmarks.** Установить целевые SLA (P50/P95/P99 для главных API).
5. **Capacity planning.** Сколько ресурсов нужно при 10 / 100 / 1000 одновременных пользователях.
6. **Multi-tenancy strategy.** Если планируется SaaS-режим — изоляция между tenant'ами.
7. **API rate limiting & quotas.** Защита от перегрузки.
8. **Internationalization completeness.** Не только UI, но и: форматы дат/чисел/валют, направление текста (RTL), сортировки по локали.
9. **Accessibility audit.** Серьёзный (а не «галочка»).
10. **Security audit.** Внешний pentest перед production.
11. **Privacy & legal review.** GDPR-compliance, terms of service, data processing agreements.
12. **Documentation for partners.** Если кто-то будет разрабатывать локализации/плагины — им нужны SDK и docs.
13. **Community support strategy.** Если open-source — как принимаем contributions, code review.
14. **Pricing & licensing model verification.** Все используемые dependencies совместимы с целевой лицензией платформы (FSL-1.1 / AGPL / MIT — что конкретно).
15. **Telemetry & product analytics** (с согласия пользователя) — какие фичи используются, где затыки.

Эти 15 пунктов — не реализация модуля, а зонтичные процессы. Их стоит запустить параллельно с модулями.
